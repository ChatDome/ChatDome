# Contributing to ChatDome

ChatDome runs security-related workflows on user machines. Changes must be explicit, reviewable, and covered by tests when behavior changes.

## Development Setup

```bash
python -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -e ./controlplane pytest
python -m pytest -q
```

For shell entrypoint changes, also run:

```bash
bash -n chatdome install.sh scripts/start.sh
```

## Pull Request Rules

- Keep each PR focused on one behavior, bug, or documentation update.
- Update README, README_ZH, config examples, and docs when user-visible behavior changes.
- Add or update tests for command routing, security checks, approval flows, and model/profile behavior.
- Do not commit secrets, local config, runtime logs, virtual environments, caches, generated bytecode, or one-shot patch scripts.
- For security-sensitive behavior, describe the risk model, user-facing failure mode, and verification scope.

## Development Standards

### Formatting and Repository Hygiene

- Store text files as UTF-8 with a final newline.
- Use LF line endings in repository files. `.editorconfig` and `.gitattributes`, when present, are the source of truth.
- Do not introduce generated cache directories such as `.pytest_cache/`, `pytest-cache-files-*/`, `__pycache__/`, build outputs, or local runtime data.
- Move reusable maintenance scripts into `scripts/maintenance/`; delete one-shot patch scripts after the change is merged.
- Keep README and README_ZH aligned for public commands, install flows, warnings, and examples.
- Add a visible README documentation entry when new long-form docs are introduced under `ChatDome-docs`.

### Code Quality Gates

Run the narrowest reliable checks for the changed surface. Use these gates when the corresponding tool is available in the repository:

| Surface | Required checks |
|---------|-----------------|
| Python behavior | `python -m pytest -q` or the targeted pytest subset |
| Python style | `ruff check .` |
| Python formatting | `ruff format --check .` |
| Shell scripts | `bash -n chatdome install.sh scripts/start.sh` and ShellCheck |
| YAML/TOML config | Parser validation or pre-commit validation |
| Package metadata | `python -m build ./controlplane` and wheel content verification |
| GitHub Actions | YAML parser validation and workflow diff review |

If a required tool is not installed locally, record that limitation in the PR or final change summary.

### Pre-Commit

When `.pre-commit-config.yaml` exists, run:

```bash
pre-commit run --all-files
```

The baseline hook set should include trailing whitespace, end-of-file fixer, YAML/TOML checks, mixed line ending checks, ruff, and ShellCheck-compatible shell validation.

### Packaging

Packaging changes must verify:

- `controlplane/pyproject.toml` has correct project metadata.
- Runtime resources are included in the wheel, including `chatdome/packs/*.yaml` and `chatdome/agent/manual/*`.
- Console scripts match the documented installation and runtime entrypoints.
- `python -m pip check` passes after editable or wheel installation.

### Release Standards

Release PRs must include:

- Version bump in the package metadata.
- CHANGELOG update under a dated release heading.
- Migration notes for config, install, command, runtime file, or security behavior changes.
- Git tag plan using `vMAJOR.MINOR.PATCH`.
- GitHub Release notes that summarize user-visible changes, compatibility notes, and verification.
- Package build verification before tag creation.

### Ownership and Maintainers

When CODEOWNERS or MAINTAINERS files exist, keep ownership current for these sensitive areas:

- Installer and update scripts
- Command execution, sandboxing, and approval flows
- Telegram command handlers
- OAuth, token, and model profile management
- Runtime paths, logs, and secret handling
- CI, packaging, and release automation

Security-sensitive files should require review from an owner before merge.

### Security-Sensitive Changes

Treat these changes as security-sensitive:

- Command execution, generated commands, sandbox validation, or approval bypass logic
- OAuth, tokens, model credentials, Telegram authorization, or admin checks
- Installer, update, service, or filesystem permission behavior
- Logging, audit files, session storage, memory, or secret redaction
- Prompt/tool definitions that can change what host commands are executed

Security-sensitive PRs must include tests or a clear manual verification plan.

## Commit Message Format

Use Conventional Commits:

```text
type(scope): summary
```

Accepted types:

| Type | Use |
|------|-----|
| `feat` | User-visible capability |
| `fix` | Bug fix |
| `docs` | Documentation-only change |
| `test` | Test-only change |
| `refactor` | Internal code structure without behavior change |
| `chore` | Tooling, CI, repository maintenance |
| `build` | Packaging or dependency change |
| `ci` | GitHub Actions or automation change |

Examples:

```text
feat(cli): add slash command completion for terminal chat
fix(auth): hide OAuth traceback behind actionable error summary
docs(cli): archive prompt_toolkit UI plan
```

The summary may be English or Chinese. Keep it direct and specific. For non-trivial changes, add a short body with the implementation scope and verification commands.

## Review Checklist

Before requesting review:

- Required tests or targeted checks pass, or limitations are documented.
- Bash syntax checks pass after shell script changes.
- Static checks, pre-commit, package build, and wheel content checks pass when configured.
- User-facing CLI and Telegram text is concise and actionable.
- README and README_ZH describe the same public commands.
- Security-sensitive changes include risk notes and verification.
- New runtime files are ignored or documented.
- Commit message follows Conventional Commits.
