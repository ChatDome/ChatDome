# Engram 记忆印迹管理

## 适用场景

用户在对话中声明了关于主机环境、行为模式、运维偏好、操作约束或拓扑的重要事实，需要判断是否应该持久化记录；或需要查询已有的 Engram 来指导当前操作。

## 核心工具

| 工具 | 用途 | 触发时机 |
|------|------|------|
| `save_engram` | 保存用户声明的事实到长期记忆 | 用户主动告知可长期复用的信息时 |
| `recall_engrams` | 查询已保存的长期记忆 | 需要了解已知约束/偏好时 |

## 判定决策树：是否应该记录

```
用户说了一句话
  │
  ├── 这是用户在主动告知信息吗？
  │     ├── 否（在提问、在发任务指令、在讨论命令输出）→ 不记录
  │     └── 是 ↓
  │
  ├── 这是关于主机环境/行为/偏好/约束/架构的持久性事实吗？
  │     ├── 否（一次性任务、临时需求、闲聊）→ 不记录
  │     └── 是 ↓
  │
  ├── 这个信息能通过执行命令直接获取吗？
  │     ├── 是（如内存大小、OS 版本、磁盘容量）→ 不记录
  │     └── 否（人类运维决策、非标准约定、组织规则）↓
  │
  └── 调用 save_engram
```

## 分类指南

| Category | 判定标准 | 应该记录的示例 | 不应该记录的示例 |
|------|------|------|------|
| `environment` | 关于主机软件栈、配置路径、版本的事实 | "防火墙用 iptables 不用 ufw"、"nginx 在 /opt/nginx" | "内存 16G"（命令可查）、"装了 docker"（命令可查）|
| `behavior` | 用户长期复用的操作方式、访问路径或处置习惯 | "用户通常通过 VPN 节点 SSH 登录服务器"、"每次发布后会检查 nginx 日志" | "刚才手动重启 nginx"（一次性事件）|
| `preference` | 用户对工具选择、操作方式的偏好 | "查日志优先用 journalctl"、"回复要简洁" | "帮我查一下日志"（任务指令不是偏好）|
| `constraint` | 用户声明的禁止/限制性规则 | "不要碰 /data 目录"、"工作时间不重启" | "现在不要重启"（临时指令不是长期约束）|
| `topology` | 网络拓扑、端口用途、上下游依赖 | "8080 是 API 网关"、"10.0.1.0/24 是内网" | "80 端口在监听"（命令可查）|

## 冲突检测与处理

调用 `save_engram` 时，系统会自动检测与已有 Engram 的冲突。

### 无冲突时

工具直接保存并返回确认信息。你必须将确认完整传达给用户：

```
🧠 已录入 Engram：[environment] 主机防火墙使用 iptables 管理，不使用 ufw (记录 ID: eg-1715100000)
```

### 检测到冲突时

工具**不保存**新记录，而是返回冲突详情。你必须向用户指出矛盾并追问：

```
⚠️ 发现与已有 Engram 冲突：
在 {时间} 你曾告诉我："{旧事实}" (记录 ID: {旧ID})
你现在说的 "{新事实}" 与此矛盾。
请确认：是否要更新为新的事实？
```

用户确认后，使用 `supersedes_id` 参数再次调用 `save_engram` 进行更新：

```
save_engram(
  category="environment",
  fact="主机防火墙改用 ufw 管理",
  source_context="用户确认已从 iptables 改用 ufw",
  supersedes_id="eg-1715100000"
)
```

## 禁止行为

| 禁止行为 | 原因 |
|------|------|
| 从命令输出自动推断并记录事实 | 命令输出是即时数据，不是用户声明的持久知识 |
| 在用户未主动声明时自行记录 | 避免记录错误或不完整的信息 |
| 记录可通过命令直接获取的客观数据 | runtime_environment 和实时命令已覆盖 |
| 记录一次性的任务指令为长期约束 | "现在不要重启"不等于"永远不要重启" |
| 检测到冲突时静默覆盖旧记录 | 必须向用户展示矛盾点并获得确认 |
| 省略录入确认通知 | 用户必须知道什么信息被写入了长期记忆 |

## `recall_engrams` 使用指南

在以下场景主动查询已知事实：

- 用户要求执行操作前，先确认是否有相关的 constraint
- 不确定主机使用什么工具/路径时，先查询 environment 和 preference
- 分析网络问题时，先查询 topology 获取已知的架构信息
- 判断登录、发布、维护等用户行为是否正常时，先查询 behavior

参数：
- `category`（可选）：按分类筛选。不填则返回所有有效记录

## 回答要求

录入事实后必须向用户发送明确的确认通知。冲突时必须展示矛盾双方的原文和时间，不要模糊描述。使用中文回答，简洁扼要。
