# 命令审批与风险解释

## 适用场景

用户询问为什么需要确认、这个命令是否安全、审批后会发生什么，或在 pending approval 状态下继续追问命令目的和风险。

## 审批流程全貌

当 AI 通过 `run_shell_command` 生成一条动态命令时，系统按以下流程处理：

```
AI 生成命令 (run_shell_command)
      │
      ▼
  静态预检 (Static Pre-check)
  ├── 写入意图检测 (has_write_intent)
  ├── 高危命令匹配 (is_critical_command)
  ├── 删除意图检测 (_has_delete_intent)
  └── 命令允许列表校验 (validate_command)
      │
      ▼
  风险分级判定
  ├── SAFE + 只读 → god_mode 下自动执行；普通模式仍需审批
  ├── UNSAFE → 进入待审批状态
  └── CRITICAL → 进入待审批状态，高危标记
      │
      ▼
  生成审批卡片（发送到 Telegram）
  用户选择：✅ 批准 / ❌ 拒绝 / 📋 查看详情
      │
      ├── ✅ 批准 → 执行命令 → 返回结果给 AI
      ├── ❌ 拒绝 → 取消执行 → AI 收到拒绝通知
      └── 📋 详情 → 触发 LLM 深度分析 → 展示详细影响评估
```

## 三级风险判定标准

| 级别 | safety_status | 判定条件 | 示例 |
|------|------|------|------|
| 安全 | `SAFE` | 纯只读查询，无系统状态变更 | `ls -la /tmp`, `cat /etc/hostname` |
| 危险 | `UNSAFE` | 存在写入、安装、配置修改意图 | `apt install ...`, `systemctl restart ...` |
| 高危 | `CRITICAL` | 不可逆或高破坏操作 | `rm -rf /`, `mkfs`, `shutdown` |

| 级别 | risk_level | 行为 |
|------|------|------|
| `LOW` | 只读操作 | god_mode 下可自动执行 |
| `MEDIUM` | 存在轻微副作用 | 需要审批确认 |
| `HIGH` | 写入/修改/安装 | 需要审批确认 |
| `CRITICAL` | 不可逆破坏 | 强制审批，高危警告 |

## god_mode（unrestricted command）差异说明

| 行为 | 普通模式 | god_mode |
|------|------|------|
| SAFE + 只读命令 | 进入待审批 | **自动执行** |
| UNSAFE 命令 | 进入待审批 | 进入待审批（仍需确认） |
| CRITICAL 命令 | 进入待审批 | 进入待审批（仍需确认） |

即使在 god_mode 下，任何被检测到变更或删除意图的命令仍然需要用户确认。

## 用户追问安全性时的标准回答结构

当用户问"这个命令安全吗"或"审批后会发生什么"时，按以下结构回答：

1. **命令目的**：一句话说明这条命令要做什么
2. **影响范围**：会影响哪些文件/服务/配置
3. **风险点**：是否会修改系统状态、是否可逆
4. **替代方案**（如有）：是否有更安全的只读替代命令
5. **建议**：明确给出"建议批准"或"建议拒绝"及理由

## 处理原则

- 明确说明待审批命令的目标、影响范围、主要风险和是否会修改系统状态。
- 不要声称待审批命令已经执行。
- 用户未批准前，不要把审批中的命令当成已产生结果来分析。
- 用户拒绝后，应停止该命令，不要尝试绕过审批。
- 审批无超时机制，命令会持续等待直到用户响应或会话结束。

## 回答要求

用简洁中文回答。必要时给出更安全的替代方案，例如先执行只读检查、缩小影响范围、备份配置或分阶段执行。
